THESIS
distribution ≠ authorization

One sentence

Fire separates who can fetch from who can read.

Distribution can be public, replicated, and cheap. Authorization is cryptographic and portable.

Implication: you can mirror ciphertext everywhere without granting access.

What changes

  • Blob hosts and relays become untrusted plumbing.
  • Access control is enforced by key possession, not server ACLs.
  • Failure mode shifts from “platform down = content gone” to “some sources down = degrade gracefully”.
Key idea: make the hard thing (authorization) small and redundant; make the big thing (bytes) mirrorable.

Slide 3

DIFFERENCE
vs HiveTalk (SFU/WebRTC)

HiveTalk / SFU model

  • Authentication: server/session-gated (JWT/OIDC/password/lobby).
  • Availability: SFU is the distribution chokepoint.
  • Media: true real-time, bi-directional, sub-second latency.
  • Trust: unless E2EE-on-top, service path can observe media.

Fire / blob + crypto model

  • Authentication: key-gated (MLS membership + NIP-17 control-plane).
  • Availability: ciphertext blobs can be mirrored across hosts.
  • Media: near-live; segmented under the hood (MSE append).
  • Trust: blob/CDN/relay can be untrusted — sees ciphertext only.

MECHANICS
what makes it real

Integrity + confidentiality

  • AES-256-GCM per chunk.
  • AAD fixed-size: SHA-256( streamId + ":" + chunkId ).
  • Prevents ciphertext swapping across streams/indices.
The blob host can serve bytes, but can’t forge valid plaintext without keys.

Authorization at scale

  • Admin is MLS controller: add/remove members.
  • NIP-17 carries join/welcome/commit control-plane messages.
  • Ongoing delivery uses MLS application messages (one publish per key payload).
  • Revocation is forward-only: stop decrypting future chunks after removal.
If a viewer desyncs (missed commit), they request resync → admin re-welcome.
