The tension we are solving

What people call decentralized

  • centralized signaling and hidden control services
  • opaque account-based permission systems
  • platform lock-in around identity and policy

What operators actually need

  • portable identity and explicit authorization flow
  • replaceable infrastructure components
  • graceful behavior under poor connectivity

Blossom Fire architecture

Identity plane

Nostr keys anchor who can request, authorize, and deliver stream control messages.

Control plane

Nostr events + NIP-17 messages coordinate access, key metadata, and stream state.

Data plane

Chunk payloads live in blob storage and are fetched independently of relay timing.

[Broadcaster] --(chunk announce)--> [Relays] [Broadcaster] --(adminkey)-------> [Admin] [Admin] ------(viewerkey)-------> [Viewer] [Viewer] -----(blob fetch)------> [Blob Store]

Protocol flow (operational view)

Step 1: Viewer requests access (access_req).
Step 2: Broadcaster emits chunk metadata + blob reference.
Step 3: Broadcaster shares admin-facing key metadata (adminkey).
Step 4: Admin forwards viewer-specific authorization (viewerkey).
Step 5: Viewer fetches blob, validates context, decodes, plays.
Step 6: Playback path chooses MSE or compatibility mode based on runtime stability.

Why this is sovereignty-positive

Operator-owned boundaries

Relay, blob service, and frontend can all be self-hosted.

Portable identity

Nostr keys, not platform accounts, define identity trust roots.

Auditable policy

Authorization is explicit message flow, not hidden backend ACLs.

Composability

Control semantics survive infrastructure swaps.

Designed for low-reliability networks

Reliability mechanisms

  • chunk-oriented delivery over fetchable blobs
  • retries for transient missing chunk windows
  • out-of-order control convergence
  • degradation instead of hard-stream collapse
Design preference

eventual continuity

over brittle realtime perfection

Current implementation reality

What works now

  • end-to-end broadcaster → admin → viewer flow
  • viewer authorization gating
  • recovery and fallback logic for browser instability
  • operationally usable prototype

Known constraints

  • Chrome MSE stability still uneven in some paths
  • compatibility mode trades smoothness for resilience
  • observability can be deeper (drift / decode telemetry)

Roadmap priorities

1. Browser pipeline hardening

fMP4/CMAF path and stricter append semantics.

2. Adaptive policies

Dynamic chunk sizing and bitrate profiles for volatility.

3. Stronger key lifecycle

Clear rotation, revocation, and viewer capability boundaries.

4. Operator controls

Sovereignty defaults exposed as first-class runtime policy.

Takeaway

Blossom Fire proves a practical point

Live media can be decentralized in architecture, sovereign in operation, and resilient on bad networks.

Not trustless hype. Concrete trust boundaries. Operator-owned control.
